# Process Ownership Responsibility Checklist

Use this checklist to ensure that process documentation is maintained, adopted, and continuously improved. The **Process Owner** is the primary responsible party unless otherwise noted.

---

## Document Management

- [ ] All process documents reside in the `docs/` directory and follow the OctoAcme naming convention
- [ ] Each document has a clear owner (Process Owner by default, unless delegated)
- [ ] Documents are versioned using git; significant changes include a meaningful commit message describing what changed and why
- [ ] Outdated or superseded sections are updated or removed promptly after a process change
- [ ] A README or index exists in `docs/` so contributors can quickly find relevant documents

---

## Process Review Cadence

- [ ] Formal process review sessions are scheduled at least quarterly (Process Owner)
- [ ] Project retrospective action items that affect processes are tracked in the process improvement backlog (Process Owner + Project Manager)
- [ ] At least one representative from each active project team is invited to process reviews (Process Owner)
- [ ] Review meeting outcomes (decisions, changes, deferred items) are documented and shared (Process Owner)

---

## Onboarding New Team Members

- [ ] New team members are pointed to `docs/` and key process documents within their first week (Process Owner + Project Manager)
- [ ] A brief walkthrough of the project management lifecycle and key roles is available or scheduled (Process Owner)
- [ ] Questions and feedback from onboarding are captured and used to improve documentation clarity (Process Owner)

---

## Process Change Management

- [ ] Proposed process changes are documented with a rationale before being adopted (Process Owner + Change Manager)
- [ ] Impact of proposed changes on current projects is assessed before rollout (Change Manager)
- [ ] All affected roles are informed of changes in advance through appropriate channels (Change Manager)
- [ ] Process documents are updated to reflect the change before or at the time of rollout (Process Owner)
- [ ] A post-change retrospective is held to assess adoption and address issues (Process Owner + Change Manager)

---

## Accountability and Adoption

- [ ] Process adherence is reviewed as part of project retrospectives (Project Manager + Process Owner)
- [ ] Deviations from defined processes are documented with rationale (Project Manager)
- [ ] Recurring deviations trigger a process review to determine whether the process should be updated (Process Owner)
- [ ] Positive process improvements identified during delivery are fed back into documentation (Project Manager + Process Owner)

---

## Continuous Improvement

- [ ] A process improvement backlog is maintained and prioritized (Process Owner)
- [ ] Team members have a clear, low-friction way to submit process improvement suggestions (Process Owner)
- [ ] Improvement actions are assigned owners and tracked to completion (Process Owner)
- [ ] Process metrics (e.g., time-to-resolve blockers, documentation freshness) are tracked where feasible (Process Owner)
